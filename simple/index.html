THE RECON FRAMEWORK

By team HEXCREW

Application that provides seamless indoor navigation using WIFI signals of that location and available sensors (Gyroscope, Accelerometer, Magnetic Field) of any basic smartphones. No additional infrastructure (extra beacon) or no known WIFI network is required (only the signal strength is measured, so the user’s smartphone need not be connected).


We implemented this in two aspects


THE RECON FRAMEWORK

 Organisations/developers can use our algorithms and modules to create custom IPS applications for their organisations based on the requirements.

PLOTSENZ

 Mobile application (built using the Recon Framework) that provides indoor navigation and LBS (Location Based Services)

IPS is not widespread like GPS as the requirement differs with case and infrastructure. By providing IPS development tools (that do not require external infrastructure) as a framework organisations/developers can build IPS applications easily


We implemented a real-time error correction ML Algorithm to filter noise in sensor data and improvised existing IPS technology to scale and Plotsenz acts as an overlay to existing GPS applications which seamlessly switches to IPS mode on entering indoors. Plotsenz universal IPS map is similar to google maps, where public contributors can scan and add locations to the map.


Problem Statement Title


Seamless indoor-outdoor navigation


Problem Statement


One of the major uses of Navigation based applications is Location-Based Services (LBS) which constitutes a significant portion of the navigation application usage. The availability of location-sensitive smartphones, with wireless broadband high-speed connections, has made low-cost applications available to a huge audience and there exists a lot of innovation in designing new types of technology and applications using these devices. Typically, it is seen that outdoor positioning is generally GPS/GNSS based. However, indoor positioning is generally based on WLAN technology. The primary reason is that GPS generally does not work for indoor applications. Uninterrupted good accuracy positioning needs thinking in the fusion of multiple technologies (Smart Phone Sensors, Wi-Fi, BLE beacons, GNSS, Dead Reckoning, etc.) to deliver a seamless outdoor and indoor positioning experience. To design/develop new ideas and mobile apps which make use of these technologies (and even other signals of opportunity) to provide seamless indoor-outdoor positioning and provide innovative location-based services.

2.Technical Details and documentation


2.1 Modules:

The Framework contains three main categories of modules, Mapping, Navigation, and Visualisation tools.


2.1.1 Mapping Modules:

Tools in this module can be used to develop applications to map locations in three different realms - WIFI, BLE, and Dead-Reckoning.


SMD3 (3D Spatial movement Data):


Mapping modules can be used to record SMD3. All mapping modules follow our framework’s standard model, handling 3d Spatial movement Data (as .SMD3 file). All modules in the framework support importing, exporting, and processing .smd3 maps.


2.1.1.1 GNSS:

GNSS-based location computation is the primary mode for outdoor navigation. All the mapping modules in the framework use Global Navigation Satellite System data in the model creation algorithms for calibrating the in-app coordinate systems. Methods in this module can also be used directly to create a GPS application



2.1.1.2 Dead reckoning:

Dead reckoning, determination without the aid of live GNSS of the position, from the record of the courses walked, the distance made (which can be estimated from velocity), the known starting point (identified using the WIFI or BLE module), and the known or estimated drift. The MEMS sensors present in most smartphones are not so accurate, they produce noise in the data which makes computing velocity and angle tricky. Also during navigation, we need to distinguish between the user moving his hands and walking. To account for these conditions, we have developed an ML-based algorithm for onboard real-time error correction. Which acts as a low pass filter to the noise in the data and ignores casual hand movements, bumping up the accuracy.



2.1.1.3 WIFI:

The WIFI module can be used to record Wifi signal strengths data models (RSSI - Received Signal Strength Information) as a .iap file, the algorithm can precisely compute the coordinates with these models when navigating. Recording the WIFI RSSI can be mapped with any wifi-enabled edge devices without actually connecting to any of those WIFI networks being scanned.


2.1.1.3.1 WIFI Repeaters:

Many companies/Malls/Institutes use WIFI Repeaters for seamless connectivity throughout their campus, WIFI SSID (Service Set IDentifier) for these repeaters will be the same, rendering the WIFI RSSI based applications useless. However in our framework, in order to account for this, we implemented additional parameters in the WIFI mapping process which involves capturing the GPS coordinates from the GNSS module (WIFI repeaters will be separated by distance on the scale of tens of meters which falls in the level where GNSS can be used to distinguish between various repeaters)


2.1.1.3.2 Signal Strength Variations Between Devices

Signal Strength readings taken at the same location with two different devices will be different, due to the phone’s WIFI antenna hardware. To take this into account we developed an algorithm (which we call “The Signal Relation Computer Algorithm”), which Based on experiments on the same, we fine-tuned the system.



The Signal Relation Computer Algorithm

Using the fact that if a particular signal is recorded with strength, attenuated by a factor of x with respect to the reference in a particular antenna, then for this given antenna, all the signals will be recorded with the same factor x



2.1.1.3.3 Portable Hotspots

When mapping to avoid polluting the model with portable hotspots (temporary mobile AP) our mapping module can differentiate between these. Mobile devices use an ipv4 range of 192.168.43.1 to 192.168.43.254 so it is possible to differentiate mobile AP from ACTUAL wifi based on this.



2.1.2 Data Collection

Self-contained MEMS sensors are an excellent option to assist the GPS for integrated navigation systems for indoor and outdoor LBS


S.No Sensor Data Collected
1 Accelerometer Acceleration of the device in X, Y, and Z-axis
2 Gyroscope Orientation and angular velocity of the device in X, Y, and Z-axis
3 GPS Latitude and Longitude of the Device
4 Magnetic-field Absolute Magnetic field strength in X Y and Z axis
5 RSSI Strength of the WIFI signal when reaching the device’s antenna
6 G-sensor Orientation of the Device

2.1.2.1 Accelerometer

An accelerometer is an electronic sensor that measures the acceleration forces acting on an object, to determine the object’s position in space and monitor the object’s movement. Acceleration, which is a vector quantity, is the rate of change of an object’s velocity (velocity being the displacement of the object divided by the change in time). We are fusing the accelerometer readings with the data from the gyroscope and G-sensorto calculate the distance travelled.



2.1.2.2 Gyroscope

A gyroscope sensor is a device that can measure and maintain the orientation and angular velocity of an object. These are more advanced than accelerometers. These can measure the tilt and lateral orientation of the object whereas an accelerometer can only measure the linear motion. We use the gyroscope readings to differentiate between walking and casual hand movements.Combined with other sensors this is used for computing the walking distances.



2.1.2.3 GPS

The Device’s GPS (Global Positioning System) is being used to get the latitude and longitude of the device for outdoor navigation and for indoors, the GPS is used for calibrating the in-app coordinate systems at equal intervals



2.1.2.4 Magnetic-field

We use the device’s onboard magnetic field sensor to measure the magnetic field strengths in the X Y and Z axis to compute the device’s angle with respect to north. This data also plays an important role in the real-time error correction module that we implemented



2.1.2.5 RSSI

The RSSI (Received Signal Strength Information) is the primary data that the WIFI and BLE modules depend on. We collect the WIFI/BLE beacon strength process it using The Signal Relation Computer Algorithm to precisely compute the location indoors.



2.1.2.6 G-sensor

We use the gravity sensor data in most of the modules for calibrating the orientations and error corrections.The iap model creation algorithm uses this data to analyze variations in the pattern extensively for improving the accuracy of the maps.



2.1.3 Mapping Methods:

We defined several standards for mapping indoors using our modules. These standard mapping methods can be followed or depending upon the problem, developers can develop their mapping standards


2.1.3.1 Wall mapping:

In wall mapping, there are two standards (default) for drawing walls. When creating an indoor map using the real-time mapping module the device being used for mapping needs to qualify the auto-config testing, which estimates the device’s quality for recording high precision 3D spatial movement data.And the module automatically configures the drawings based on the user’s movement as follows.


  1. While walking in the anticlockwise direction, the wall will be drawn to the right of the reference pointer.
  2. While walking in the clockwise direction, the wall will be drawn to the left of the reference pointer

In this mode the user needs to walk close to the wall tracing its perimeters. Works with closed polygons (concave and convex) and open structures (buildings/structures that are not closed)


Data from this module will have metadata for the online editor to recognise this as wall mapping data, which will be reflected in the rendering process. (elevation in z-axis and shadows in rendering)


2.1.3.2 Path mapping:

Similar to wall mapping methods, we have defined standards for mapping walkable areas. Path mapping requires the same specs as wall mapping. The process is also similar, the user who is mapping should walk through the Campus dt holding the smartphone pointing the way they walk.


The intersection points in the path will be recognised by the module, and path joint nodes will be added automatically which will be later used in the navigation, for path computation to the destination. These nodes can be added or edited manually with the map editor (more will be discussed in their respective sections).


The image shown here is the render of an actual path mapping data (Rendered using recon Framework’s spatial movement visualisation tools)


2.1.3.3 Node Mapping:

There are two main categories of node clusters in our IPS maps,Des_NodesandNav_Node these nodes are used for locating destinations and navigation.


Des_Nodes:

Des_Node are markers that can be placed over the map to identify rooms/locations on the map. It can contain location names, notes/descriptions,s, and constraints. When mapping the path, des_Nodes can be added, later in the online map editor, the user can update the name and other parameters.


Nav_Node:

Nav_Nodes are auto-generated by the algorithms at junctions and turnings. This can also be added manually. These nav_nodes are used for finding the connecting path between the user’s current location and the destination Des_node.



2.1.3.4 WIFI / BLE mapping:

In WIFI / BLE mapping, We receive RSSI information continuously in intervals of time to generate the .iap model and draw the map. This happens simultaneously with other mappings (Path mapping and wall Mapping).



Rendering smd3 (3D Spatial movement Data):

Using this framework, developers can create applications that can visualise Spatial movement data paths. The rendering modules support all types of spatial movement data collected using any of the mapping modules from the recon framework because they all follow the same standards of spatial movement mapping defined by us in the recon framework.


2D rendering:

A 2D path can be generated in three different ways: common Parameter: Time period (tracing path in that particular time period)


# Mode Description
1 Top view projection 2D render of the path viewed from the top, no differentiation between different levels(stories/floors)
2 Cross-section (parameters: z_index, threshold) Renders path in the x-y plane at z = z_index with tolerance threshold as threshold
3 Isometric Projection Renders the map in isometric mode
4 Custom Render view from custom point/angle

3D rendering:

3D path is rendered from the .SMD3 (Spatial Movement Data) is similar to 2D rendering but completely interactive - the following operations can be done on the generated model:


# Action Detail

Web: Javascript -for building web applications.

Navigation:(Shortest Path, All possible Paths)

Navigation modules convert .iap files to graphs by processing them after combining them with the node data. When initialising it uses latitude and longitude from GPS to fetch the nearest iap data from the server, following which it computes the starting position by measuring the wifi signals and processing it with the relative signal strength algorithm.



Shortest Path Shortest path between the starting from the user’s location, covering the least number of nodes possible.
Best Route (Multiple nodes) If the user needs to visit multiple nodes in a trip, this method can be used to get the best route covering all the given nodes.
All Possible Path To get all the possible paths to reach the destination node - gives details of the route (info like using stairs/ elevator/ escalator, etc)


Map Editing:

We have made our advanced online map editor with which users can edit sync their map scans to the universal indoor model and with this editor they can also create a custom IPS application (specifically made for an organisation) Users can add names, description to the nodes in the map, can manually add new nodes by simply drag and dropping new node from the side panel Users can also export their maps as image/smd3 files from this site directly. All their contribution will be linked to their accounts. Developers can modify this template to build their custom map editor based on requirements. The codes are well documented
